# Profile claims need a chain of evidence

The cost of producing a tailored CV went to zero last year. So did the signal. A recruiter screening a hundred applications can't tell which were written by a person who knows the job and which were spun up by a language model that read the job description ten seconds ago. Both sides know this. Both sides are now using language models to compensate. The information asymmetry that hiring relied on — that producing a thoughtful, specific, well-targeted claim was costly enough to be a signal — is gone.

The question isn't "how do we detect AI-generated CVs." The detection arms race is unwinnable, and even if it weren't, the underlying problem isn't the artefact. The problem is that every claim in a profile is now cheap, and we have nothing structural that distinguishes claims that should be believed from claims that shouldn't.

## The reframe

Profile claims need a chain of evidence. Not proof of identity — you already control your GitHub account, that's not the gap. Proof of *claims*: "I know Rust" should link, in the profile itself, to public Rust code you've written, where anyone can click through and see for themselves. "I worked at Acme" should sit next to verifiable artefacts of that work. "I'm an inventor on patent X" should resolve to the patent office. Every assertion in your profile should be one click away from its independent source.

This isn't a new idea, exactly. Keybase did it for identity proofs a decade ago. Awesome-* lists treat curated repos as a knowledge substrate. Web-of-trust systems have been gesturing at this shape for thirty years. What's new is that the *cost of producing unverifiable claims* has collapsed at the same time the *infrastructure for verifying them* has matured. GitHub is queryable. Package registries are queryable. Patent offices have APIs. The pieces exist; they just haven't been assembled into a profile format.

The name is gist.id. The gist of who you are, with the evidence attached. The product is a profile at `gist.id/<your-github-handle>` where every skill that can be cross-checked against a public source is, and where the rest is honestly marked as a claim with no evidence.

## How it works

The whole pipeline is static files, public Git, GitHub Pages, GitHub Actions, and one Cloudflare Worker. There is no platform server, no database, no account system.

**Your profile is a public GitHub repo named `gist-id`.** It's created from a template and contains a handful of markdown files — `profile.md`, `resume/work.md`, `resume/skills.md`, and so on. The layout is opinionated and documented. You can start fresh, or import from a LinkedIn data export — the builder reads the Positions, Education, Projects, and Patents CSVs from LinkedIn's full archive and writes the corresponding markdown for you. It skips Skills (LinkedIn's skill data is partly autogenerated and not worth importing), and ignores everything private (connections, messages, ad targeting).

**The builder is a Rust binary, wrapped in a composite GitHub Action.** On every push, and on a 6-hour cron, the Action runs the builder against your repo. The builder parses the markdown, queries the GitHub API for verification evidence, signs the output with your ed25519 key (stored as a repo secret), and writes `feed.postcard` to GitHub Pages. The signed feed is the wire format: compact, deterministic, and verifiable.

**Verification today queries two paths.** First, all your owned public non-fork repos. Second, every repo you've pushed to in the last 90 days, via the public events API. For each, we fetch the language byte breakdown from `/repos/{owner}/{repo}/languages`. If a language matches a skill name in your `skills.md` (case-insensitive), the skill gets a ✓ badge linking to the repo with the most bytes in that language. A tooltip shows all repos that contributed to the match. If we find languages you haven't claimed, the profile surfaces them as suggestions.

**The edge worker is a Cloudflare Worker written in Rust + Leptos 0.8.** It compiles to WASM and runs on Workers. When a request hits `gist.id/<handle>`, the Worker fetches `<handle>.github.io/gist-id/feed.postcard`, deserialises (it peeks the schema version first and renders a friendly "rebuilding" page on mismatch), and SSRs the profile. The rendering is fully synchronous — no reactive runtime, no hydration shenanigans, just `view! { ... }.to_html()` via Leptos's Tachys engine. The schema crate compiles to both native (for the builder) and WASM (for the worker), so the types are the same end-to-end.

**There are two layers of caching.** The Worker's outgoing subrequests to GitHub Pages use Cloudflare's `cf_properties` with `cache_everything: true` and a 1-hour TTL, overriding GitHub's own 10-minute `Cache-Control`. The Worker's own response is then cached in `Cache::default()` keyed by a normalised URL, with the same 1-hour TTL (5 minutes for 404s). On a warm cache, the Worker doesn't run; on a cold response cache but warm subrequest cache, the Worker runs but doesn't touch GitHub Pages.

**Discovery is a separate indexer.** A second Rust binary, run by a 6-hour-offset cron in GitHub Actions, queries GitHub's repo search for `topic:gist-id` and `name:gist-id` (the convention is that profile repos are tagged with the `gist-id` topic). For each result, it fetches the signed feed, verifies the signature, extracts the verified skills, and writes per-tag postcard files plus a profiles directory to a separate `gist-id/index` repo. That repo is itself served from GitHub Pages. The edge Worker fetches from there to render `gist.id/` (landing page with tag directory) and `gist.id/?tag=rust` (engineers with verified Rust evidence). The index is verified-only: claims without evidence don't appear in discovery, which preserves the signal of the search.

The mechanism for joining the network is just: tag your repo with `gist-id`, sign a feed. Removing yourself is: delete the topic. The "registry" is GitHub's search index. There is no central database to leave.

## What we actually verify (and what we don't)

This is the honest scope of v0.1:

- ✓ Programming-language skills, by repo byte breakdown across your owned + recently-pushed-to repos
- ✓ ed25519 signatures, verified by the indexer before a profile is included in discovery
- ✓ GitHub handle is structurally tied to repo ownership — controlling the repo *is* the identity

What we don't do yet:

- We don't read commit content. We see "this user pushed to a repo where 60% of the bytes are Rust." We don't grade your code, and we can't tell whether you wrote the Rust or someone else did.
- We don't verify against package registries. A crates.io / npm / PyPI cross-check would be a much stronger signal than commits — published artefacts are higher trust than pushed-to repos — but it's not built yet.
- We don't catch sophisticated fakes. You could AI-generate a public Rust repo, push it, and the verification would pass. The signal would still be there for a viewer who clicked through — an AI-generated repo with no stars, no users, no history beyond a single weekend, looking suspiciously like an LLM training-set average — but the gist.id page wouldn't catch it automatically.
- We don't re-verify at render time. The Worker trusts the signed cached numbers. Independent re-verification is a planned post-MVP enhancement.
- We don't verify private work. Your best code might be at $employer, in a private repo. We can't see it. We won't pretend.

These are deliberate constraints. The property we wanted is *anyone can re-verify the claim, in seconds, against an authoritative public source*. The cost of that property is that a lot of real work is invisible — it's the same trade-off open source has always made. Making the chain of evidence stronger — package registries, patent inventor checks, identity cross-references via Mastodon `rel="me"` and personal sites — is the obvious next move.

## Try it

My profile is at [gist.id/FigmentEngine](https://gist.id/FigmentEngine). The example data is real; the verification on my Rust and JavaScript skills resolves to actual repos. Set yours up at [gist.id/getting-started](https://www.gist.id/getting-started) — about 30 minutes if you edit the markdown directly, or 10 minutes plus a 24-hour wait if you use the LinkedIn import path. Source for the whole stack, including the Rust workspace, the edge Worker, and the indexer, is at [github.com/gist-id](https://github.com/gist-id).

The discovery side: [gist.id/?tag=rust](https://gist.id/?tag=rust) shows engineers with verified Rust evidence. Right now that's mostly me; the network is exactly one week old.

## A note on what built this

I wrote this in a week. With Claude. Which reveals the irony of skills-based hiring in a gen-AI world: I'm shipping a system that says profile claims need a chain of evidence, while leaning hard on an AI that produces claim-shaped text on demand.

The point isn't that AI is bad. The point is that the cost of *producing claims* has gone to zero, so whatever value is left in hiring has to migrate to *verification*. Things that can be cheaply cross-checked against an independent public source still have signal. Things that can't, don't. The interesting design question — and the question I think the next few years of hiring will be wrestling with — is how to construct evidence chains that hold up when the cost of fabricating the unverified part has collapsed.

I don't know the full answer. I suspect it involves more chains of evidence, not fewer, and weaker assumptions about what a "skill" claim implies. This MVP is me feeling out the shape of it. If you have a better mechanism, I'd like to hear about it.
